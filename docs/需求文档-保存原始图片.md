# 需求文档：报警时同步保存原始图片

## 1. 背景与动机

### 1.1 现状

当前系统在触发报警时，保存的图片是经过以下处理的**标注帧**（annotated frame）：

1. 绘制边界投影线（红/黄/橙三道防线）
2. 绘制目标检测框（矩形 + 类别名 + 置信度）
3. 绘制 "AI" 徽章 + 时间戳水印

这些标注叠加在原始画面上，虽然便于人工审核，但也存在以下问题：

- **标注遮挡原始画面**：检测框、边界线、水印等会遮挡关键区域的细节
- **事后分析受限**：无法对原始画面做二次检测、特征提取或不同算法的比对
- **取证完整性不足**：在某些审计/合规场景下，需要保留未经任何处理的原始画面作为证据链的一环

### 1.2 目标

在保存报警图片（标注帧）的同时，**额外保存一帧对应的原始画面**（无任何标注），两者通过文件名关联，便于后续追溯和分析。

## 2. 功能需求

### 2.1 核心需求

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-01 | 报警触发时，同时保存标注帧和原始帧 | P0 |
| FR-02 | 原始帧与标注帧使用相同的目录结构，存放在 `raw/` 子目录下 | P0 |
| FR-03 | 原始帧文件名与标注帧文件名保持关联（共享时间戳和 rule_id） | P0 |
| FR-04 | 原始帧的图片质量（JPEG quality）与标注帧完全一致 | P0 |
| FR-05 | 原始帧**不上传** MinIO，仅本地保存 | P0 |
| FR-06 | 原始帧**不上报**平台 API | P0 |
| FR-07 | Suppressed 图片（未上报的告警）**不需要**保存原始帧 | P0 |
| FR-08 | 保存失败时不影响原有标注帧的保存流程（容错降级） | P0 |

### 2.2 文件命名与目录结构

#### 当前结构
```
./res/{stream_name}/{YYYYMMDD}/
├── alerts/
│   └── alert_{rule_id}_{YYYYMMDD-HHmmss}.jpg
├── suppressed/
│   └── suppressed_{reason}_{YYYYMMDD-HHmmss}_track{id}.jpg
└── clips/
    └── clip_{target_id}_{YYYYMMDD-HHmmss}.mp4
```

#### 新增结构
```
./res/{stream_name}/{YYYYMMDD}/
├── alerts/
│   └── alert_{rule_id}_{YYYYMMDD-HHmmss}.jpg              ← 标注帧（不变）
├── raw/
│   └── alert_{rule_id}_{YYYYMMDD-HHmmss}_raw.jpg          ← 原始帧（新增）
├── suppressed/
│   └── suppressed_{reason}_{YYYYMMDD-HHmmss}_track{id}.jpg ← 不变，无原始帧
└── clips/
    └── clip_{target_id}_{YYYYMMDD-HHmmss}.mp4
```

#### 命名规则
- 原始帧文件名 = 标注帧文件名 + `_raw` 后缀
- 示例：
  - 标注帧：`alert_alarm_any_detection_20260515-151055.jpg`
  - 原始帧：`alert_alarm_any_detection_20260515-151055_raw.jpg`

### 2.3 原始帧的定义

**原始帧** = 摄像头拉流解码后的原始画面，即 `reference_frame`（`camera.py:1622`）。

以下内容**不包含**在原始帧中：
- ❌ 边界投影线（红/黄/橙）
- ❌ 检测框和标签
- ❌ "AI" 徽章和时间戳水印

原始帧**包含**以下内容（来自摄像头本身的画面）：
- ✅ 摄像头 OSD 叠加的时间戳（如有）
- ✅ 摄像头本身的画面增强（如有）

### 2.4 上传与上报策略

| 项目 | 标注帧 | 原始帧 |
|------|--------|--------|
| 本地保存 | ✅ | ✅ |
| MinIO 上传 | ✅ | ❌ 不上传 |
| 平台上报 | ✅ | ❌ 不上报 |
| 图片质量 | JPEG 95 | JPEG 95（与标注帧一致） |

## 3. 技术方案

### 3.1 数据流变更

当前调用链（`camera.py` → `alert_system.py`）：

```
camera.py:1627  confirmed_frame = reference_frame.copy()
                ... (绘制标注) ...
camera.py:1646  return {'frame': confirmed_frame, ...}
                ↓
alert_system.py:182  _save_alert_image(event, frame_for_alert)   ← 只保存标注帧
```

**变更后**：将 `reference_frame`（原始帧）一并传递到 alert_system：

```
camera.py:1627  confirmed_frame = reference_frame.copy()
                ... (绘制标注) ...
camera.py:1646  return {'frame': confirmed_frame, 'raw_frame': reference_frame, ...}
                ↓
alert_system.py:182  _save_alert_image(event, frame_for_alert, raw_frame)
                     _save_raw_image(event, raw_frame)           ← 新增：保存原始帧
                     ↑ 不触发 MinIO 上传，不触发平台上报
```

### 3.2 涉及修改的文件和方法

| 文件 | 方法 | 变更内容 |
|------|------|---------|
| `camera.py` | `_process_infer_results()` | 在返回的 dict 中增加 `raw_frame` 字段 |
| `camera.py` | `_process_frame()` | 透传 `raw_frame` 到 `process_frame_alerts()` |
| `alert/alert_system.py` | `AlertHandler.handle_alert()` | 接收并传递 `raw_frame` |
| `alert/alert_system.py` | `AlertHandler._save_alert_image()` | 新增 `raw_frame` 参数，调用 `_save_raw_image()` |
| `alert/alert_system.py` | **新增** `_save_raw_image()` | 将原始帧保存到 `raw/` 目录 |

### 3.3 关键实现细节

#### `_save_raw_image()` 方法（新增）

```python
def _save_raw_image(self, event: AlertEvent, raw_frame: np.ndarray) -> str:
    """保存原始帧（无标注）到 raw/ 子目录，不上传、不上报"""
    if raw_frame is None:
        return ""
    try:
        ts_epoch = float(getattr(event, 'timestamp', time.time()) or time.time())
        ts = time.strftime("%Y%m%d-%H%M%S", time.localtime(ts_epoch))
        fname = f"alert_{event.rule_id}_{ts}_raw.jpg"
        path = os.path.join(self._ensure_output_dir("raw", ts_epoch), fname)
        cv2.imwrite(path, raw_frame)  # 与标注帧使用相同的默认 JPEG 参数
        logging.info(f"原始图片已保存: {path}")
        return path
    except Exception as e:
        logging.warning(f"保存原始图片失败（不影响告警流程）: {e}")
        return ""
```

#### 图片质量一致性

- 标注帧通过 `cv2.imwrite(path, frame)` 保存，使用 OpenCV 默认 JPEG 质量 95
- 原始帧使用相同的 `cv2.imwrite(path, raw_frame)` 调用，参数完全一致
- 如需显式指定质量，可使用 `cv2.imwrite(path, raw_frame, [cv2.IMWRITE_JPEG_QUALITY, 95])`

#### 容错策略

- 原始帧保存失败**不阻断**标注帧保存流程
- 原始帧保存失败**不阻断**后续的视频片段拼接、MinIO 上传、平台上报流程
- 使用 `try/except` 包裹，失败时仅记录 warning 日志

### 3.4 配置项

在 `config/config.json` 的 `alarm` 节点中新增：

```json
{
  "alarm": {
    "save_raw_image": true
  }
}
```

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `alarm.save_raw_image` | bool | `true` | 是否在报警时同步保存原始帧 |

## 4. 资源影响评估

### 4.1 存储空间

- 每张原始帧与标注帧尺寸相同（摄像头分辨率，如 1920×1080）
- JPEG 质量 95 时，单张约 300KB-500KB
- 假设每天 100 次报警：额外增加约 **30MB-50MB/天**
- 假设每天 1000 次报警：额外增加约 **300MB-500MB/天**

### 4.2 性能影响

- `reference_frame.copy()` 已在现有流程中执行（`camera.py:1627`），原始帧可直接引用，**无需额外拷贝**
- 增加一次 `cv2.imwrite()` 调用，单帧写入耗时约 5-15ms（SSD）
- 对整体报警处理延迟影响可忽略（当前已有 2-3 次图片写入操作）

### 4.3 内存影响

- 原始帧通过引用传递，不额外分配内存
- 生命周期与标注帧相同，无内存泄漏风险

| 编号 | 测试场景 | 预期结果 |
|------|---------|---------|
| TC-01 | 触发一次报警 | `alerts/` 目录下有标注帧，`raw/` 目录下有原始帧，文件名关联 |

| TC-06 | Suppressed 场景 | 仅有 `suppressed/` 下的标注帧，无 `raw/` 下的原始帧 |
| TC-07 | 原始帧保存失败 | 标注帧正常保存，日志中有 warning，后续流程不受影响 |
| TC-08 | 配置关闭 `save_raw_image` | 仅保存标注帧，无 `raw/` 目录生成 |
